## 翻訳ワークフロー

### 全体フロー

```
1. 準備
   ↓
2. 翻訳
   ↓
3. セルフレビュー
   ↓
4. ピアレビュー（オプション）
   ↓
5. 検証
   ↓
6. コミット
   ↓
7. 進捗更新
```

### フェーズ別ワークフロー

#### Phase 1: 準備

**目的**: 翻訳作業の準備を整える

**手順**:
1. 用語集（GLOSSARY.md）を確認
2. スタイルガイド（TRANSLATION_STYLE_GUIDE.md）を確認
3. 既存の翻訳済みドキュメントを参照
4. 翻訳対象ファイルの構造を理解

**成果物**:
- 翻訳対象ファイルのリスト
- 特殊な用語や表現のメモ

**所要時間**: 10-15分/ファイル

#### Phase 2: 翻訳

**目的**: ドキュメントを日本語に翻訳する

**手順**:
1. セクションごとに翻訳
   - 見出しから開始
   - 段落を1つずつ翻訳
   - コードブロックのコメントを翻訳
2. 用語集に従って用語を統一
3. 自然な日本語表現を心がける
4. リンクとフォーマットを維持

**注意点**:
- 一度に全体を翻訳しようとしない
- セクションごとに完成させる
- 不明な用語は用語集を確認

**成果物**:
- 翻訳済みドキュメント（初稿）

**所要時間**: 1-2時間/ファイル（サイズによる）

#### Phase 3: セルフレビュー

**目的**: 自分で翻訳の品質を確認する

**手順**:
1. 翻訳から時間を置く（可能なら1時間以上）
2. 品質チェックリストを使用して確認
3. 以下の観点でレビュー：
   - 技術的正確性
   - 日本語の自然さ
   - 用語の統一性
   - フォーマットの正確性
4. 必要に応じて修正

**チェックリスト**:
- [ ] 技術的な意味が正確に伝わっている
- [ ] 自然な日本語表現になっている
- [ ] 用語集に従っている
- [ ] リンクが機能する
- [ ] コードブロックが正しく表示される

**成果物**:
- レビュー済み翻訳ドキュメント

**所要時間**: 30-45分/ファイル

#### Phase 4: ピアレビュー（オプション）

**目的**: 他の翻訳者からフィードバックを得る

**手順**:
1. レビュアーに翻訳を共有
2. フィードバックを受け取る
3. フィードバックに基づいて修正
4. 修正内容をレビュアーに確認

**レビュー観点**:
- 技術的正確性
- 日本語の自然さ
- 用語の統一性
- スタイルガイドへの準拠

**成果物**:
- レビュー済み翻訳ドキュメント
- レビューコメント

**所要時間**: 20-30分/ファイル（レビュアー）

#### Phase 5: 検証

**目的**: 技術的な正確性を確認する

**手順**:
1. リンク検証
   ```bash
   # リンクチェックツールを使用（将来実装）
   python scripts/validate_links.py <file>
   ```

2. フォーマット検証
   ```bash
   # Markdownリンターを使用
   markdownlint <file>
   ```

3. コードブロック検証
   - コード例が正しく表示されるか確認
   - コメントが適切に翻訳されているか確認

**成果物**:
- 検証済み翻訳ドキュメント
- 検証レポート（問題がある場合）

**所要時間**: 10-15分/ファイル

#### Phase 6: コミット

**目的**: 翻訳をリポジトリにコミットする

**手順**:
1. 変更をステージング
   ```bash
   git add <translated-file>
   ```

2. コミットメッセージを作成
   ```bash
   git commit -m "docs(i18n): translate <file> to Japanese"
   ```

3. プッシュ
   ```bash
   git push origin <branch>
   ```

**コミットメッセージ規約**:
```
docs(i18n): <action> <file> to Japanese

例:
docs(i18n): translate README.md to Japanese
docs(i18n): update glossary with new terms
docs(i18n): fix typos in design.md
```

**成果物**:
- Gitコミット

**所要時間**: 5分/ファイル

#### Phase 7: 進捗更新

**目的**: 翻訳進捗を記録する

**手順**:
1. tasks.mdのチェックボックスを更新
2. translation_status.yaml（将来実装）を更新
3. 次のタスクを確認

**成果物**:
- 更新された進捗記録

**所要時間**: 5分/ファイル

## 役割と責任

### 翻訳者

**責任**:
- ドキュメントの翻訳
- セルフレビューの実施
- 用語集とスタイルガイドへの準拠
- 進捗の報告

**必要なスキル**:
- 日本語と英語の読み書き能力
- 技術文書の理解力
- Markdown知識
- Git基本操作

### レビュアー（オプション）

**責任**:
- 翻訳の品質確認
- フィードバックの提供
- 用語の統一性確認

**必要なスキル**:
- 翻訳者と同等のスキル
- レビュー経験

### プロジェクトマネージャー（オプション）

**責任**:
- 全体進捗の管理
- リソースの調整
- 品質基準の維持

## 品質保証プロセス

### 品質基準

翻訳は以下の基準を満たす必要があります：

1. **正確性**: 技術的な意味が正確に伝わる
2. **自然さ**: 日本語として自然で読みやすい
3. **一貫性**: 用語と表現が統一されている
4. **完全性**: 翻訳漏れがない
5. **フォーマット**: Markdown構文が正しい

### チェックリスト

#### 内容の正確性
- [ ] 技術的な意味が正確に伝わっている
- [ ] 重要な情報が欠落していない
- [ ] コード例が正しく動作する
- [ ] 数値やバージョン番号が正確

#### 日本語の品質
- [ ] 自然な日本語表現になっている
- [ ] 文法的に正しい
- [ ] 読みやすい文章構造
- [ ] 適切な文体（敬体/常体）

#### 用語の統一性
- [ ] 用語集に従っている
- [ ] プロジェクト内で統一されている
- [ ] 技術用語が適切に使用されている
- [ ] カタカナ表記が統一されている

#### フォーマット
- [ ] Markdown構文が正しい
- [ ] リンクが機能する
- [ ] コードブロックが正しく表示される
- [ ] テーブルが正しく表示される
- [ ] 画像が正しく表示される

#### 構造
- [ ] 元の構造が維持されている
- [ ] 目次が正しく更新されている
- [ ] セクション番号が正しい
- [ ] 階層構造が適切

### 検証方法

#### 1. 手動検証

**リンクチェック**:
```bash
# 各リンクをクリックして確認
# 内部リンク、外部リンク、アンカーリンク
```

**表示確認**:
```bash
# Markdownプレビューで確認
# コードブロック、テーブル、画像
```

#### 2. 自動検証（将来実装）

**リンク検証**:
```bash

```

**フォーマット検証**:


**用語統一性チェック**:
```bash

```

## 進捗管理

### 進捗トラッキング

#### tasks.md

翻訳タスクは `.kiro/specs/documentation-i18n-japanese/tasks.md` で管理します。

**フォーマット**:
```markdown
- [ ] 未着手
- [-] 進行中
- [x] 完了
```

**更新方法**:
```markdown
# タスク開始時
- [ ] → - [-]

# タスク完了時
- [-] → - [x]
```

#### translation_status.yaml（将来実装）

詳細な進捗情報を記録します。

**フォーマット**:
```yaml
phase_2:
  status: in_progress
  files:
    - path: necrocode/dispatcher/README.md
      status: completed
      translator: kiro
      reviewed: true
      completed_at: "2025-12-06"
```

### 進捗レポート

#### 週次レポート（オプション）

**内容**:
- 完了したファイル数
- 進行中のファイル数
- 残りのファイル数
- 課題と対策

**フォーマット**:
```markdown
## 週次進捗レポート (2025-12-06)

### 完了
- necrocode/dispatcher/README.md
- necrocode/agent_runner/README.md

### 進行中
- necrocode/artifact_store/README.md (50%)

### 次週の予定
- necrocode/artifact_store/README.md 完了
- necrocode/review_pr_service/README.md 開始
```

## トラブルシューティング

### よくある問題と解決策

#### 問題1: 用語の訳語が不明

**症状**: 適切な日本語訳が分からない

**解決策**:
1. GLOSSARY.mdを確認
2. 既存の翻訳済みドキュメントを検索
3. 技術用語辞書を参照
4. 不明な場合は英語のまま使用し、後で確認

#### 問題2: 文章が不自然

**症状**: 直訳すぎて日本語として不自然

**解決策**:
1. 意味を理解してから翻訳
2. 日本語として自然な表現に言い換え
3. TRANSLATION_STYLE_GUIDE.mdの例を参照
4. 時間を置いて再度読み直す

#### 問題3: リンクが壊れる

**症状**: 翻訳後にリンクが機能しない

**解決策**:
1. リンクテキストのみ翻訳、URLは変更しない
2. 相対パスが正しいか確認
3. アンカーリンクの場合、見出しIDを確認

#### 問題4: コードブロックが崩れる

**症状**: コードブロックが正しく表示されない

**解決策**:
1. バッククォート（```）の数を確認
2. 言語指定が正しいか確認
3. インデントが崩れていないか確認

#### 問題5: 進捗が遅い

**症状**: 予定より翻訳が遅れている

**解決策**:
1. 1ファイルずつ完成させる
2. 完璧を求めすぎない（後で修正可能）
3. 難しいセクションは後回し
4. 休憩を取る

## ベストプラクティス

### 効率的な翻訳

1. **セクションごとに完成させる**
   - 全体を一度に翻訳しない
   - 1セクション完成したらコミット

2. **用語集を活用する**
   - 翻訳前に用語集を確認
   - 不明な用語は追加

3. **既存の翻訳を参照する**
   - 同じ表現を再利用
   - 一貫性を保つ

4. **時間を置いてレビュー**
   - 翻訳直後はミスに気づきにくい
   - 1時間以上置いてから確認

### 品質向上

1. **声に出して読む**
   - 不自然な表現に気づきやすい
   - リズムを確認

2. **他の人に読んでもらう**
   - 客観的な意見を得る
   - 理解しやすさを確認

3. **継続的に改善**
   - フィードバックを記録
   - 次回に活かす

## 付録

### A. コミットメッセージ例

```bash
# 新規翻訳
docs(i18n): translate dispatcher README to Japanese

# 更新
docs(i18n): update glossary with dispatcher terms

# 修正
docs(i18n): fix typos in agent runner README

# 用語統一
docs(i18n): unify terminology in repo pool docs
```

### B. レビューコメント例

```markdown
## レビューコメント

### 良い点
- 技術用語が正確に翻訳されている
- 自然な日本語表現

### 改善点
- L45: 「実行することができます」→「実行できます」
- L67: 用語「Task」の訳が不統一（タスク/課題）

### 質問
- L89: この表現で技術的に正確ですか？
```

### C. 進捗レポートテンプレート

```markdown
## 翻訳進捗レポート (YYYY-MM-DD)

### サマリー
- 完了: X ファイル
- 進行中: Y ファイル
- 残り: Z ファイル
- 進捗率: XX%

### 今週の完了
- [ ] ファイル1
- [ ] ファイル2

### 次週の予定
- [ ] ファイル3
- [ ] ファイル4

### 課題
- 課題1: 説明
- 課題2: 説明

### 備考
その他の情報
```
