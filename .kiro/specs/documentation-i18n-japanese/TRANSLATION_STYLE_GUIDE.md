## 基本方針

### 1. 自然な日本語

**直訳を避け、日本語として自然な表現を使用します。**

❌ **悪い例**:
```
このシステムはタスクを実行することができます。
```

✅ **良い例**:
```
このシステムはタスクを実行します。
```

### 2. 技術的正確性

**技術的な意味を正確に伝えます。**

❌ **悪い例**:
```
データベースに情報を保存します。（曖昧）
```

✅ **良い例**:
```
データベースにタスク情報を永続化します。（具体的）
```

### 3. 読みやすさ

**専門家でない読者にも理解しやすい表現を心がけます。**

❌ **悪い例**:
```
本システムは非同期処理機構を実装しており、並行実行制御を行う。
```

✅ **良い例**:
```
このシステムは複数のタスクを同時に実行できます。
```

### 4. 一貫性

**プロジェクト全体で統一された用語と表現を使用します。**

- 用語集（GLOSSARY.md）を必ず参照
- 既存の翻訳済みドキュメントと表現を揃える
- 同じ概念には同じ訳語を使用

## 文体とトーン

### ドキュメントタイプ別の文体

| ドキュメントタイプ | 文体 | 特徴 | 例 |
|------------------|------|------|-----|
| **README** | 敬体（です・ます調） | 親しみやすく、歓迎的 | 「NecroCodeへようこそ！このガイドでは...」 |
| **仕様書** | 常体（だ・である調） | 正確で、技術的 | 「システムはタスクを管理する。」 |
| **ガイド** | 敬体（です・ます調） | 手順が明確、実用的 | 「次の手順でインストールします。」 |
| **リファレンス** | 常体（だ・である調） | 簡潔で、網羅的 | 「このメソッドはタスクを作成する。」 |

### トーンの統一

**プロフェッショナルでありながら親しみやすいトーンを維持します。**

✅ **推奨**:
- 「〜できます」「〜します」（能動的）
- 「〜してください」（丁寧な指示）
- 「〜することをお勧めします」（提案）

❌ **避ける**:
- 「〜すべきである」（命令的すぎる）
- 「〜かもしれません」（不確実すぎる）
- 「〜だと思います」（主観的すぎる）

## 用語の扱い

### 1. 技術用語

**用語集（GLOSSARY.md）に従って統一された訳語を使用します。**

| 英語 | 日本語 | 備考 |
|------|--------|------|
| Task | タスク | ❌「課題」「作業」は使わない |
| Workspace | ワークスペース | ❌「作業領域」は使わない |
| Repository | リポジトリ | ❌「保管庫」は使わない |

### 2. プロジェクト固有用語

**初出時は英語併記、以降は日本語のみ使用します。**

✅ **初出時**:
```
Spirit（スピリット）は、NecroCodeのAIエージェントです。
```

✅ **2回目以降**:
```
スピリットはタスクを並列実行します。
```

### 3. そのまま英語を使用する用語

**以下は英語のまま使用します：**
- API, CLI, JSON, YAML
- Git, GitHub, Docker, Kubernetes
- Pull Request (PR), Commit, Branch

### 4. カタカナ表記のルール

**長音符（ー）の使用**:
- 3音以上の語尾の長音は省略しない
  - ✅ サーバー、ユーザー、マネージャー
  - ❌ サーバ、ユーザ、マネージャ

**外来語の表記**:
- 一般的なカタカナ表記に従う
  - ✅ コンピューター、データベース
  - ❌ コンピュータ、データベイス

## 文章構造

### 1. 文の長さ

**1文は40〜60文字程度を目安にします。**

❌ **長すぎる例**:
```
NecroCodeは、自然言語のジョブ記述を完全に実装されたアプリケーションに変換する、
AI駆動のマルチスピリット開発フレームワークで、専門化されたAIスピリットのチームを
召喚し、ソフトウェアプロジェクトの設計、実装、テスト、デプロイを自律的に協働して
行います。
```

✅ **適切な例**:
```
NecroCodeは、AI駆動のマルチスピリット開発フレームワークです。自然言語のジョブ記述を
受け取り、完全に実装されたアプリケーションに変換します。専門化されたAIスピリットの
チームが、設計、実装、テスト、デプロイを自律的に協働して行います。
```

### 2. 箇条書き

**複数の項目を列挙する場合は箇条書きを使用します。**

❌ **悪い例**:
```
このシステムはタスク管理、ワークスペース管理、アーティファクト管理の機能を提供します。
```

✅ **良い例**:
```
このシステムは以下の機能を提供します：

- タスク管理
- ワークスペース管理
- アーティファクト管理
```

### 3. 段落構成

**1段落1トピックの原則を守ります。**

✅ **良い構成**:
```
## インストール

NecroCodeをインストールするには、Pythonが必要です。

次のコマンドでインストールできます：
[コマンド]

インストール後、設定ファイルを作成します。
```

### 4. 見出しの階層

**見出しは論理的な階層構造を保ちます。**

```markdown
# レベル1: ドキュメントタイトル
## レベル2: 主要セクション
### レベル3: サブセクション
#### レベル4: 詳細項目
```

## コードとコマンド

### 1. コードブロック

**コメントのみ日本語化、コード自体は変更しません。**

❌ **間違い**:
```python
def タスクを作成(名前):
    """タスクを作成する"""
    return タスク(名前)
```

✅ **正しい**:
```python
def create_task(name):
    """タスクを作成する"""
    return Task(name)
```

### 2. インラインコード

**技術用語や変数名は `バッククォート` で囲みます。**

✅ **例**:
```
`Task` クラスは、タスクの状態を管理します。
`workspace_manager.py` ファイルを編集してください。
```

### 3. コマンド例

**コマンド自体は変更せず、説明コメントを日本語化します。**

✅ **例**:
```bash
# 依存関係をインストール
pip install -r requirements.txt

# テストを実行
pytest tests/
```

### 4. 出力例

**出力例も適切に日本語化します。**

✅ **例**:
```
実行結果:
✓ タスク1: 完了
✓ タスク2: 完了
✗ タスク3: 失敗
```

## フォーマットとレイアウト

### 1. Markdown構文

**元のMarkdown構文を維持します。**

- 見出し: `#`, `##`, `###`
- リスト: `-`, `*`, `1.`
- リンク: `[テキスト](URL)`
- 強調: `**太字**`, `*斜体*`
- コードブロック: ` ```言語 `

### 2. リンク

**リンクテキストは日本語化、URLは変更しません。**

❌ **間違い**:
```markdown
[See documentation](README.md)
```

✅ **正しい**:
```markdown
[ドキュメントを参照](README.md)
```

### 3. テーブル

**ヘッダーと内容を日本語化します。**

✅ **例**:
```markdown
| 機能 | 説明 | ステータス |
|------|------|-----------|
| タスク管理 | タスクの作成と実行 | 完了 |
| ワークスペース管理 | ワークスペースの管理 | 進行中 |
```

### 4. 画像とダイアグラム

**代替テキストを日本語化します。**

✅ **例**:
```markdown
![システムアーキテクチャ図](architecture.png)
```

## よくある間違いと修正例

### 1. 直訳しすぎ

❌ **悪い例**:
```
あなたはこのコマンドを実行することができます。
```

✅ **良い例**:
```
次のコマンドを実行します。
```

### 2. 冗長な表現

❌ **悪い例**:
```
タスクを実行することが可能です。
```

✅ **良い例**:
```
タスクを実行できます。
```

### 3. 受動態の多用

❌ **悪い例**:
```
タスクはシステムによって実行されます。
```

✅ **良い例**:
```
システムがタスクを実行します。
```

### 4. 不自然なカタカナ語

❌ **悪い例**:
```
このフィーチャーはユーザビリティをエンハンスします。
```

✅ **良い例**:
```
この機能は使いやすさを向上させます。
```

### 5. 曖昧な指示代名詞

❌ **悪い例**:
```
タスクとワークスペースを作成します。それを実行します。
```

✅ **良い例**:
```
タスクとワークスペースを作成します。タスクを実行します。
```

## 翻訳プロセス

### ステップ1: 準備
1. 用語集（GLOSSARY.md）を確認
2. 既存の翻訳済みドキュメントを参照
3. 元のドキュメントの構造を理解

### ステップ2: 翻訳
1. セクションごとに翻訳
2. 用語集に従って用語を統一
3. 自然な日本語表現を心がける

### ステップ3: レビュー
1. 技術的正確性を確認
2. 日本語の自然さを確認
3. フォーマットを確認

### ステップ4: 検証
1. リンクが機能することを確認
2. コードブロックが正しく表示されることを確認
3. 品質チェックリストを実行
